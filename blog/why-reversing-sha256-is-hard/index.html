Why Reversing Sha256 Is Hard

17 Aug, 2025

A sha256 digest is the concatination of 8 32 bit integers represented in hexadecimal form. The digest for the letter ‘a’

ca978112ca1bbdcafac231b39a23dc4da786eff8147c4e72b9807785afee48bb

can also be represented like this in psuedo-code

h0 := 0xca978112 
h1 := 0xca1bbdca
h2 := 0xfac231b3
h3 := 0x9a23dc4d
h4 := 0xa786eff8
h5 := 0x147c4e72
h6 := 0xb9807785
h7 := 0xafee47bb

Let’s focus on just reversing h0. how did we arrive at this number? The sha256 calcualtes h0 as the addition of the previous value of h0 with a value a. What was the previous value of h0? We have no idea! We’ve already hit a road block before we’ve began!

We can try simplifying the problem. Assume the message is small enough to fit in to 512 bits, this is the size of a block that sha256 works on. Since h0 + a happens once per block, h0 would be its initial starting value: 0x6a09e667

This let’s us turn out attention on to how a was calculated:

func rightRotate(value uint32, shift uint) uint32 {
	return (value >> shift) | (value << (32 - shift))
}

// the 64 iteration of the sha256 algorithm, which gives the final value of `a`
s1 := rightRotate(e, 6) ^ rightRotate(e, 11) ^ rightrotate(e, 25)
ch := (e & f) ^ ((^e) & g)

temp1 := h + s1 + ch + uint32(k[63]) + uint32(w[63])

s0 := rightRotate(a, 2) ^ rightRotate(a, 13) ^ rightRotate(a, 22)
maj := (a & b) ^ (a & c) ^ (b & c)
temp2 := s0 + uint32(maj)

a = temp1 + temp2

We’ve stumbled in to a endless maze of misdirection! We have to know all of the previous values of a, b, c, d, e, f, g and h along with the 32 bit value in the message schedule w at index 63. Ignoring the 8 variables a..g, Let’s consider finding just the value of w[63]. That was originally calculated as:

s0 := rightRotate(w[48], 7) ^ rightRotate(w[48], 18) ^ (w[48] >> 3)
s1 := rightRotate(w[61], 17) ^ rightRotate(w[61], 19) ^ (w[61] >> 10) 
w63 = w[47] + s0 + w[56] + s1

and following the specification of the algorithm, to know w[48] we need to know w[33], and to know w[33] we need to know w[18], and to know w[18], we need to know w[2] which is potentially a 32 bit chunk of the original message!

It doesn’t end here, even if we guessed these values, we would have to guess them 63 more times and hope that values h0...h8 align with their initially instantiated values. Remember, we’ve simplified this so that only 1 block is involved. There could be multiple blocks!

This leads to a second strength of the sha256 algorithm. Not only is it hard to reverse, it’s difficult to find other messages that would result in the same hash, i.e, a collision.
